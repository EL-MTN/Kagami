# Configuration

All env vars are declared in `packages/shared/src/env.ts` (the `@kagami/env`
spec — schema + docs in one place). `packages/shared/src/config.ts` keeps the
runtime contract: `config` is parsed eagerly at module load (exit-on-invalid,
cross-field rules skipped so the dashboard's transitive import through the
`@kokoro/db` barrel never enforces bot-only pairings), and `validateConfig()`
— called at bot startup — runs the eight cross-field rule groups
(provider→key pairings, the Kao both-or-neither pair, TTS/STT/image format
rules, Browserbase-in-cloud-mode, MCP name uniqueness, BlueBubbles pairings).

`.env` lives at `apps/bot/.env`. The table below, `apps/bot/.env.example`,
and `apps/bot/turbo.json` are **generated** from the spec — edit `env.ts` and
run `npm run env:gen`; `npm run env:check` fails on drift.

Notes:

- Blank values are treated as unset and kept values are trimmed, uniformly
  across every key (this replaces the per-var preprocess the old config.ts
  hand-applied to only some keys). A blank on a defaulted var falls back to
  the default.
- `KAO_TOKEN` enforces the workspace-wide ≥16-char floor and `KAO_URL` must
  be a host-only origin — the same rules Kizuna and Kao itself enforce.
- `LOG_PRETTY` (force pino-pretty; unset = TTY-detect) is a runner-level
  knob, not part of the schema — `dev-all.sh` exports `LOG_PRETTY=1` for all
  dev children.
- The dashboard reads its own `.env` (Next.js auto-load); the schema above
  governs the bot process and everything imported through `@kokoro/shared`.
